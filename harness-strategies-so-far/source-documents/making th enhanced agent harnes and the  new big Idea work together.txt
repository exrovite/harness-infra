## The Key Integration: State Machine Enforcement

The `run-harness.sh` outer loop now drives the OpenClaw state machine instead of our simpler phase model:

```bash
# The harness enforces these transitions deterministically:
# REQUESTED → CONTRACT_LOCKED → IMPLEMENTED → VERIFIED → ACCEPTED
#                                    ↓
#                                  STUCK
#                          (escalate → guided → resume)

case $STATE in
  "REQUESTED")
    # Planner agent creates task.contract.json
    # Harness validates contract exists and is locked
    # Harness creates features.json and tests.json
    # → CONTRACT_LOCKED
    ;;
    
  "CONTRACT_LOCKED")
    # Generator agent implements, TDD-first
    # Harness runs detect-loop.sh → STUCK if triggered
    # Harness validates: all features "passing", zero test failures
    # Builder CANNOT advance past here — harness checks
    # → IMPLEMENTED
    ;;
    
  "IMPLEMENTED")
    # Layer 2 deterministic checks first (protocol compliance)
    # Ensure dev server running
    # Evaluator agent runs 7-layer verification:
    #   Layer 1: Evidence before verdict (no reading builder's code)
    #   Layer 2: Structured evidence chain per assertion
    #   Layer 3: Adversarial framing
    #   Layer 4: Calibration gate (plant failure, verify caught)
    #   Layer 5: Independent environment
    #   Layer 6: Cross-examination (second prompt)
    #   Layer 7: Immutable evidence archive
    # → VERIFIED (or back to CONTRACT_LOCKED with feedback)
    ;;
    
  "VERIFIED")
    # Notify you on Telegram with evidence bundle
    # Wait for your sign-off
    # → ACCEPTED
    ;;
    
  "STUCK")
    # Harness auto-detected (loop detector or escalation triggers)
    # Notify Telegram with facts, not theories
    # Block until your response
    # Inject guidance, resume from pre-STUCK state
    ;;
    
  "ACCEPTED")
    # Git checkpoint, session summary, update memory manifest
    # Done
    ;;
esac
```
