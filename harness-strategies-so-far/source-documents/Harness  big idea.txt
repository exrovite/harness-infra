First Principles Decomposition
Let me break this down to why the agent falls off the loop in the first place. There are exactly five fundamental failure modes, and every solution needs to address all of them.

Failure Mode 1: Context Window Saturation

This is the root cause most people miss. Claude's context window is a shared public good. Your CLAUDE.md, your skills, the files Claude reads, the conversation history, the tool call results — all of it competes for the same space. The research from HumanLayer's analysis confirms that Claude Code's system prompt alone contains roughly 50 individual instructions. That's nearly a third of the instruction-following capacity consumed before your protocol even loads.

When context fills up, compaction happens. During compaction, Claude summarises the conversation and drops detail. Your carefully sequenced protocol steps? They get compressed. The agent "forgets" it was on step 4 of 7 because step 4's specifics got compacted away.

From the cognitive systems angle: this is identical to what happens when you overload a clinician's working memory during a complex case formulation. The maintaining cycle diagram you drew two sessions ago gets fuzzy. You start skipping steps in the protocol. Not because the protocol is bad, but because there's too much competing for attention.

Failure Mode 2: No External State Tracking

Here's the fundamental architectural problem. The LLM is stateless between turns. It has no persistent memory of "I'm on step 4." It infers its position from context each time. That inference is probabilistic, not deterministic.

The claw-code system solved this with clawhip — an external daemon that tracked state outside the agent's context window. The agent didn't need to remember where it was. The system told it.

Pedro Sant'Anna's academic workflow (which the research surfaced as one of the most mature implementations) does the same thing: agents communicate through standardised files on disk — JSON, CSV, logs — not through hidden conversational state. Every step is inspectable and rerunnable.

Failure Mode 3: No Phase Isolation

The Wiggum CLI documentation makes this distinction sharply. A basic Ralph loop — just running while true; do cat PROMPT.md | claude; done — is a single undifferentiated blob. The agent does everything in one pass. If something goes wrong, you restart from scratch.

The structured Ralph loop separates planning from implementation from testing from verification. Each phase has its own inputs, outputs, and success criteria. Each phase can fail independently.

RIPER (Research → Innovate → Plan → Execute → Review) goes further — it actually blocks the agent from writing code until it's past the Research and Plan phases. The agent literally cannot edit files during the Research phase. That's deterministic enforcement, not hopeful instruction.

Failure Mode 4: No Verification Gate

Boris Tane's workflow, which the research flagged as one of the most effective single-developer approaches, has one core principle: never let Claude write code until you've reviewed and approved a written plan. He annotates the plan file directly, Claude reads the annotations, updates the plan, and only proceeds to code when the plan passes.

Most people skip this. They let the agent plan and execute in the same breath. The result is that errors compound. A bad line in a plan creates many bad lines of code. A bad line in a research phase that misunderstands the system creates many bad lines in the plan, which creates exponentially more bad code.

Failure Mode 5: No Deterministic Operations

Here's the insight from Matthew Groff's implementation guide: never send an LLM to do a linter's job. Deterministic operations — running tests, checking formatting, validating schemas, tracking status — should be bash scripts, not LLM inferences. Every time you ask the LLM to "check if the tests pass," you're burning tokens and introducing non-determinism into something that should be binary.

The CCPM framework takes this furthest: status, standup, search, and validation all run as bash scripts. Fast, consistent, no LLM token cost.

# Master Development Protocol

## BEFORE ANYTHING ELSE
1. Read `.claude/state/current-phase.json`
2. Read `.claude/state/checkpoint.md`
3. You are resuming from the phase and step indicated. Do NOT restart.

## PHASE SEQUENCE (follow in order, never skip)

### PHASE 1: RESEARCH
- Read the project spec/requirement
- Read existing codebase structure
- Identify what exists, what's missing, what conflicts
- OUTPUT: Write findings to `.claude/state/research-output.md`
- GATE: Do not proceed until research-output.md exists and is complete
- Run: `bash .claude/scripts/checkpoint.sh RESEARCH`

### PHASE 2: PLAN
- Read research-output.md
- Read available strategies in `.claude/protocols/[project]-protocol.md`
- Creatively combine strategies using first principles
- OUTPUT: Write plan to `.claude/plans/plan.md` with numbered tasks and acceptance criteria
- GATE: Check if `.claude/plans/plan-approved.md` exists

  - If YES: Proceed to Phase 3
- Run: `bash .claude/scripts/checkpoint.sh PLAN`

### PHASE 3: EXECUTE
- Read plan-approved.md
- Execute tasks one at a time in order
- After each task: update current-phase.json with task number
- After each task: run `bash .claude/scripts/validate.sh`
- If validation fails: fix before proceeding
- OUTPUT: Working code that passes all validation
- Run: `bash .claude/scripts/checkpoint.sh EXECUTE`

### PHASE 4: REVIEW
- Spawn reviewer subagent: `@reviewer`
- Reviewer reads plan-approved.md and compares against implementation
- Reviewer writes findings to `.claude/state/review-output.md`
- If ISSUES FOUND: return to relevant step in Phase 3
- If CLEAN: proceed
- Run: `bash .claude/scripts/checkpoint.sh REVIEW`

### PHASE 5: VERIFY
- Run `bash .claude/scripts/validate.sh --full`
- Run any project-specific test suites
- Compare output against acceptance criteria in plan-approved.md
- OUTPUT: Write verification report to `.claude/state/verify-output.md`
- If FAIL: return to Phase 3 with specific failure details
- If PASS: proceed
- Run: `bash .claude/scripts/checkpoint.sh VERIFY`

### PHASE 6: COMPLETE
- Run `bash .claude/scripts/notify.sh "Task complete — verification passed"`
- Commit with descriptive message
- Update current-phase.json to COMPLETE

## RULES (never violate)
- NEVER skip a phase
- NEVER proceed past a GATE without meeting the gate condition
- ALWAYS write state to disk before and after each phase
- ALWAYS run validation after code changes
- If you lose track of where you are: read current-phase.json

Component 4: The Ralph Loop with Phase Isolation

This is where you wire it all together for autonomous operation. Instead of a basic Ralph loop that restarts everything, you use a phase-aware loop:

bash
#!/bin/bash
# run-protocol.sh
MAX_ITERATIONS=30
ITERATION=0

while [ $ITERATION -lt $MAX_ITERATIONS ]; do
  PHASE=$(cat .claude/state/current-phase.json | jq -r '.phase')
  
  if [ "$PHASE" = "COMPLETE" ]; then
    echo "Protocol complete."
    exit 0
  fi
  
  if [ "$PHASE" = "PLAN" ] && [ ! -f .claude/plans/plan-approved.md ]; then
    echo "Waiting for human approval. Check Telegram."
    sleep 60
    continue
  fi
  
  claude --permission-mode auto-accept -p \
    "Read .claude/protocols/master-protocol.md and continue from current phase. \
     Current state is in .claude/state/current-phase.json"
  
  ITERATION=$((ITERATION + 1))
done
This gives you the Ralph loop's persistence but with the phase isolation that prevents the "undifferentiated blob" problem. Each iteration picks up where the last left off because the state lives in files, not in the conversation.

Let me connect this back to the first principles:

Context window saturation → solved by progressive disclosure. The master protocol is short. Phase-specific instructions load only when needed. State lives in files, not in conversation. Subagents get their own context windows for review tasks.

No deterministic operations → solved by pushing everything that can be a script into .claude/scripts/. Status tracking, validation, notification, checkpointing — all bash. Only the creative work (planning, coding, reviewing) uses the LLM.

Anthropic found this was so severe with Sonnet 4.5 that compaction alone couldn't fix it — they needed full context resets. The Real Problem: Instruction Decay Under Context Pressure
When you load a skill file — say it's 200 lines of detailed protocol — the agent reads it. In that moment, it has the full picture. But here's what happens next:

The agent starts working. Each tool call, each file read, each code generation step adds to the context. Your 200-line protocol is now competing with 500 lines of code output, 300 lines of test results, 150 lines of error messages. The protocol instructions are still technically in context, but they're now a small signal in a lot of noise. The model's attention to those specific instructions degrades proportionally.

This is different from context saturation (where things get compacted away) and different from context anxiety (where the agent rushes to finish). This is instruction dilution — the protocol is still there, but the agent stops attending to it because more recent, more salient information is demanding its attention.

From the clinical protocol angle, you already know this pattern. A trainee therapist reads a session protocol before seeing a client. The protocol says: assess risk, set agenda, review homework, introduce technique, assign new homework, summarise. The trainee starts the session, the client says something unexpected, the trainee follows that thread, and forty minutes later they realise they never assessed risk or reviewed homework. The protocol was in their head. They just stopped attending to it under the pressure of live interaction.

The fix in clinical training isn't "read the protocol harder." It's structured fidelity checks — someone observing the session with a checklist, tapping them on the shoulder at transition points. That's what's missing from the architecture.

What Needs To Change: Protocol Fidelity Enforcement
Here's what I'd add to the architecture — a mechanism that forces the agent to check back against its protocol at defined intervals, and that catches negative loops (where the agent keeps trying the same failing approach instead of applying a known solution).

The Fundamental Distinction: Soft vs. Hard Enforcement
Your developer drew a clean line that I kept blurring. Let me make it explicit:

Soft enforcement = instructions in markdown files that the LLM reads and (hopefully) follows. Works when context is fresh. Degrades under pressure. Is essentially advisory.

Hard enforcement = bash scripts, Claude Code hooks, and harness logic that runs outside the LLM. Works regardless of context pressure. Is actually deterministic.

Harness enforcement = the outer loop that controls what the agent sees, when sessions start and stop, and whether the agent is allowed to proceed. The agent doesn't participate in these decisions.

My previous architecture was heavy on soft, thin on hard, and missing the harness layer almost entirely. Here's what it should actually look like.

#!/bin/bash
# .claude/scripts/verify-fix-applied.sh (revised — no eval)

FIX_FILE=".claude/state/next-fix.md"
[ ! -f "$FIX_FILE" ] && exit 0

PASS=true
CURRENT_TYPE=""
CURRENT_FILE=""
CURRENT_PATTERN=""
CURRENT_BEFORE=""

while IFS= read -r line; do
  [[ -z "$line" || "$line" == \#* ]] && continue
  
  case "$line" in
    *"type: file_exists"*)
      CURRENT_TYPE="file_exists" ;;
    *"type: file_contains"*)
      CURRENT_TYPE="file_contains" ;;
    *"type: test_passes"*)
      CURRENT_TYPE="test_passes" ;;
    *"file:"*)
      CURRENT_FILE=$(echo "$line" | sed 's/.*file: //') ;;
    *"pattern:"*)
      CURRENT_PATTERN=$(echo "$line" | sed 's/.*pattern: //') ;;
    *"before_pattern:"*)
      CURRENT_BEFORE=$(echo "$line" | sed 's/.*before_pattern: //')
      
      # Execute the accumulated check
      case "$CURRENT_TYPE" in
        "file_exists")
          if [ ! -f "$CURRENT_FILE" ]; then
            echo "FAIL: $CURRENT_FILE does not exist"
            PASS=false
          fi ;;
        "file_contains")
          if [ -n "$CURRENT_BEFORE" ]; then
            # Check ordering: pattern must appear before before_pattern
            PATTERN_LINE=$(grep -n "$CURRENT_PATTERN" "$CURRENT_FILE" 2>/dev/null | head -1 | cut -d: -f1)
            BEFORE_LINE=$(grep -n "$CURRENT_BEFORE" "$CURRENT_FILE" 2>/dev/null | head -1 | cut -d: -f1)
            if [ -z "$PATTERN_LINE" ] || [ -z "$BEFORE_LINE" ] || [ "$PATTERN_LINE" -gt "$BEFORE_LINE" ]; then
              echo "FAIL: Pattern ordering wrong in $CURRENT_FILE"
              PASS=false
            fi
          else
            if ! grep -q "$CURRENT_PATTERN" "$CURRENT_FILE" 2>/dev/null; then
              echo "FAIL: Pattern not found in $CURRENT_FILE"
              PASS=false
            fi
          fi ;;
        "test_passes")
          # Only allow pre-approved test commands
          ALLOWED_CMDS=("pytest" "npm test" "cargo test" "python -m unittest")
          CMD=$(echo "$line" | sed 's/.*command: //')
          CMD_BASE=$(echo "$CMD" | awk '{print $1}')
          ALLOWED=false
          for allowed in "${ALLOWED_CMDS[@]}"; do
            [[ "$CMD_BASE" == "$allowed" || "$CMD" == "$allowed"* ]] && ALLOWED=true
          done
          if [ "$ALLOWED" = true ]; then
            timeout 120 bash -c "$CMD" > /dev/null 2>&1
            if [ $? -ne 0 ]; then
              echo "FAIL: Test command failed: $CMD"
              PASS=false
            fi
          else
            echo "SKIP: Command not in allowlist: $CMD"
          fi ;;
      esac
      # Reset for next check
      CURRENT_TYPE="" CURRENT_FILE="" CURRENT_PATTERN="" CURRENT_BEFORE=""
      ;;
    *"command:"*)
      # Handle test_passes type
      CMD=$(echo "$line" | sed 's/.*command: //')
      CMD_BASE=$(echo "$CMD" | awk '{print $1}')
      ALLOWED=false
      ALLOWED_CMDS=("pytest" "npm test" "cargo test" "python -m unittest")
      for allowed in "${ALLOWED_CMDS[@]}"; do
        [[ "$CMD_BASE" == "$allowed" || "$CMD" == "$allowed"* ]] && ALLOWED=true
      done
      if [ "$ALLOWED" = true ]; then
        timeout 120 bash -c "$CMD" > /dev/null 2>&1
        [ $? -ne 0 ] && echo "FAIL: $CMD" && PASS=false
      fi
      CURRENT_TYPE="" ;;
  esac
done < <(sed -n '/## Verify/,/## /p' "$FIX_FILE")

[ "$PASS" = false ] && exit 1
rm "$FIX_FILE"
exit 0
Three check types — file_exists, file_contains (with optional ordering via before_pattern), and test_passes (allowlisted commands only, with timeout). No arbitrary shell execution.

#!/bin/bash
# .claude/scripts/run-claude-safe.sh
# Wrapper that handles crashes, rate limits, retries

PROMPT_FILE="$1"
MODE="${2:-auto-accept}"
MAX_RETRIES=3
RETRY_DELAY=60
ATTEMPT=0

while [ $ATTEMPT -lt $MAX_RETRIES ]; do
  cat "$PROMPT_FILE" | claude --permission-mode "$MODE"
  EXIT_CODE=$?
  
  if [ $EXIT_CODE -eq 0 ]; then
    exit 0
  fi
  
  ATTEMPT=$((ATTEMPT + 1))
  
  if [ $ATTEMPT -lt $MAX_RETRIES ]; then
    bash .claude/scripts/notify-telegram.sh "💥 Session failed (exit $EXIT_CODE). Retry $ATTEMPT/$MAX_RETRIES in ${RETRY_DELAY}s."
    sleep $RETRY_DELAY
    RETRY_DELAY=$((RETRY_DELAY * 2))  # Exponential backoff
  fi
done

# All retries exhausted
bash .claude/scripts/write-handoff.sh "SESSION_CRASH_FINAL"
bash .claude/scripts/notify-telegram.sh "🛑 Session failed $MAX_RETRIES times. State saved. Agent paused."
echo "BLOCKED" > .claude/state/agent-blocked.md
echo "Claude session failed repeatedly. Last exit code: $EXIT_CODE" >> .claude/state/agent-blocked.md
exit 1

Great question. Let me be straight about this because the answer has two parts and the distinction matters.

How Claude Code Actually Scopes Configuration
Claude Code reads configuration from three levels, and they cascade:

Level 1 — User-level (global, applies to every session on your machine): Lives in ~/.claude/ on your machine. Anything here applies to every Claude Code session you open, regardless of which project folder you're in. This is where global settings, hooks, and agents go.

Level 2 — Project-level (per repository): Lives in .claude/ at the root of a specific git repository. Only applies when you're working inside that repo. This is where project-specific agents, skills, and commands go.

Level 3 — Conversation-level (per session): What you type, what gets loaded dynamically. Ephemeral.

What This Means For The Harness
Most of what we've built is infrastructure — bash scripts, hooks, the Telegram daemon, the harness loop, the crash recovery system. This infrastructure doesn't need to be copied into every project. It can live at the user level and work globally.

About Telegram Specifically
You said "especially if I am not working in Telegram." This is important — the Telegram bridge is optional, not structural. It's one notification and input channel. The harness works without it. Here's what changes:

Without Telegram: Every point where the harness would notify you and wait for a response, it instead writes a file and pauses. You check the state files manually when you come back to your machine. The wait-for-human.sh script watches for human-response.md regardless of how that file gets created — you could write it by hand, via a simple local web UI, or even a VS Code extension that watches for blocked states and gives you a dialog box.

With Telegram: You get push notifications on your phone and can reply from anywhere. Your reply becomes human-response.md. That's the only integration point.

With WhatsApp (via Twilio): Same architecture, different API endpoint in notify.sh. The transport is interchangeable.
